## Part 1 — what happened

The session resumed a completed-but-unverified spec/plan corpus (119 pieces, spec + plan each) and ran it through review, decision-making, and packaging — **not** implementation. No platform code was written; the deliverables are review artifacts and a durable decision ledger.

### Phase 1 — Handoff discovery & state assessment
Started from `_meta/HANDOFF.md`. A cheap state check (file counts, byte-size scan for stubs) showed the gap-fill the handoff worried about was **already complete** — 119+119 files, zero stubs, `plan-ADR-0041.md` full. So item 1 of the handoff was a non-issue; the real remaining work was verification (item 2), adversarial review (item 3), and the Operational/NFR layer (item 5).

### Phase 2 — 13-agent fan-out
Dispatched a single wave of 13 subagents, model-matched to task type per the user's directive (Opus for red-team/authoring judgment, Sonnet for csv analysis, Haiku for mechanical existence checks): 5 T0 red-team reviews (R1–R5), 6 Operational/NFR authoring agents (OPS1–OPS6), 1 consistency-existence check, 1 csv-reciprocity analysis. Agents wrote to disk and returned brief reports; the orchestrator committed centrally (subagents ran no git) to avoid index races. Most auto-backgrounded; results arrived as notifications.

### Phase 3 — consolidator timeout, recovered by hand
A 14th subagent tasked with consolidating ~100 KB of review files into a decision register **timed out at ~28 minutes with no output**. Rather than re-dispatch, the orchestrator wrote `DECISIONS-NEEDED.md` directly from the agent reports already in context. Lesson banked (see §3.4).

### Phase 4 — area-split packaging
Per a user choice, the single working branch was split into three clean PRs by area — #12 (OPS layer), #13 (reviews + registers), #14 (consistency + 4 csv fixes) — disjoint file sets, conflict-free. PR #10 (the original working branch) was closed as superseded with an explanatory comment. All three merged.

### Phase 5 — the jargon rejection
The first decision register was written in engineer shorthand ("the 4 approval events that A23/B5/B10/A14 bind to…"). The user rejected it flatly: *"written for an AI, not a human."* A Sonnet agent rewrote it into `DECISIONS-NEEDED-PLAIN.md` with a "cast of characters" and every code spelled out. This reframed all subsequent communication.

### Phase 6 — interactive decision walkthrough
The bulk of the session: working through D-01…D-11, Part-2 design choices #1–#27, and quality nits QN-01…QN-05 one cluster at a time. The user corrected course repeatedly — on scope, on jargon, on decisions-vs-complaints, and on specific architecture (auth model, memory abstraction, quota schema, Envoy, observability). Each ruling was recorded immediately in `_meta/reviews/DECISIONS-LOG.md` and pushed.

### Phase 7 — the Crossplane v1/v2 discovery (the headline finding)
A grep prompted by the user's "stop using 'claim'" correction revealed the corpus is pervasively written on the **Crossplane v1** resource model (~270 "claim"/`X`-prefix references, codified in **ADR-0041**), while the platform is **Crossplane v2** (claims removed). Verified the adjacent claim — that the Kubernetes agent-sandbox could replace Envoy — by fetching its API docs (`NetworkPolicySpec` is L3/L4 only), which **kept** Envoy in the design. This became D-11, the one remaining open decision and the next session's main job.

### Phase 8 — scope correction & wrap
The user flagged that the OPS layer (and my register) had crept into baseline/platform/cluster concerns. Tracing it: the OPS subagents had invented out-of-scope requirements, which I piped into the register unfiltered. Most of OPS1–6 turned out to be out-of-scope or future. Recorded the consequence, produced `OPEN-DECISIONS.md`, updated the handoff for a fresh session, wrote a verbatim hand-off prompt, and merged everything.

## Part 4 — proposed ADRs

Architectural decisions made this session that the user may want to record formally. **Titles only — no specs.** All are detailed in `_meta/reviews/DECISIONS-LOG.md`; the next session is already slated to author several. Use the `adr` skill to write any of them.

- **Crossplane v2 resource model** — supersedes ADR-0041; namespaced composite resources, no claims, no `X`-prefix duality. *(Already planned for the next session.)*
- **Single owner per CloudEvent namespace** — exactly one component authors each `platform.*` namespace's schemas; others declare an explicit dependency (no co-ownership).
- **Agent credential model** — `system` (system-process creds, not tenant-specific) vs `system-mediated` (platform-held, per-tenant-isolated in LiteLLM); agents never hold credentials; system agents use workload identity.
- **Secrets abstracted by External Secrets Operator** — agent-os consumes ESO from the baseline; operator-entered creds flow LiteLLM GUI → k8s Secret → ESO PushSecret; KMS/rotation/at-rest are out of scope.
- **Two-layer egress control** — Envoy enforces L7 (FQDN/method) for external egress; the agent-sandbox `NetworkPolicy` provides the L3/L4 default-deny baseline.
- **Tenant quota model** — extensible typed `quotas` list on `TenantOnboarding`; admission requires `cpu`+`memory` *presence* (value may be explicit `unlimited`); native `ResourceQuota`/`LimitRange` enforcement.
- **Security events on the bus** — every component, on detecting a security event, also emits it under `platform.security` (schema owned by OPA); consumers subscribe via the bus.
- **Observability alerts via the event bus** — AlertManager publishes to `platform.observability`; consumers (HolmesGPT et al.) subscribe; no direct point-to-point alert wiring.
- **Policy-bundle governance** — security-team-owned global bundle; signed + verified at load; audit-mode soak then enforce, gated by the promotion pipeline.
