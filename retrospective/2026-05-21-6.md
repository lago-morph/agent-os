## Part 1 — what happened

### Phase 1 — capability probe

The user asked whether subagents can run on a different inference model than the parent. The answer was confirmed inline from the `Agent` tool schema: it accepts an optional `model` parameter taking `opus | sonnet | haiku`. No code was touched in this phase — it was establishing the surface area for the work that followed.

### Phase 2 — joke test + identity probe

The user asked for a head-to-head: dispatch a Haiku subagent with "tell me a joke", produce a joke myself, compare, and try to independently verify the subagent was actually running Haiku.

The interesting result was on the identity check. The Haiku subagent self-reported `claude-haiku-4-5-20251001`, but I had it dump environment variables and the `ANTHROPIC_MODEL` env var was **empty**. None of the `CLAUDE_CODE_*` vars expose the model in use. So inside the harness there is no environment-variable ground truth — the strongest available identity signal is the subagent's own self-report (sourced from its system prompt) plus, indirectly, observed capability. That finding became the basis for an agents-file suggestion.

### Phase 3 — three-way capability benchmark

To go beyond a joke (which doesn't discriminate models well), I designed a graduate-level algebraic topology problem with a definite right answer and a known sharp failure mode: compute π₂(S¹ ∨ S²) as a ℤ[π₁]-module. The naïve wrong answer (universal cover = ℝ ∨ S², trivial t-action, π₂ ≅ ℤ) is qualitatively different from the correct answer (universal cover has one S² per integer, π₂ ≅ ℤ[t, t⁻¹] free of rank 1).

Three subagents dispatched in parallel in a single message (default = Opus, Sonnet, Haiku):

| Model | Result | Duration |
|---|---|---|
| Opus 4.7 (default) | Correct, tight | 20.2s |
| Sonnet 4.6 | Correct, verbose with explicit self-correction | 32.9s |
| Haiku 4.5 | **Wrong** — exactly the predicted failure mode (cover = ℝ ∨ S², trivial π₁ action, π₂ ≅ ℤ) | 13.0s |

This was a clean discriminating result: the predicted failure mode for Haiku showed up almost verbatim. Capability ordering matched expectation (Opus ≥ Sonnet ≫ Haiku for this problem); latency ordering matched expectation (Haiku fastest, Sonnet slowest because it produced the most tokens).

### Phase 4 — formalize as `benchmark-subagent` skill

The user asked to package this workflow into a skill. Key design decisions during the skill write-up:

- **Trigger design** — the user wanted breadth without false positives. The user's verbal spec included "any word containing 'agent'", which read literally would match nearly every Claude Code session. Re-parsed as "any word containing 'agent' **in close proximity to a benchmark-synonym**" — that's tractable. The skill description encodes this conjunctively: benchmark-synonym (benchmark, compare, evaluate, test, race, shootout, bake-off, head-to-head, A/B, rank, "stack up", "pit against", vs, versus) near target-synonym (model, LLM, Claude, Opus, Sonnet, Haiku, AI, or any word with "agent" as substring). Plus explicit negative triggers: "which model should I use in production" (recommendation, not benchmark), agent-architecture discussion, HTTP user-agent.
- **Default problems** — the user said "two graduate-level homework problems in different fields" if none specified. Used the already-validated topology problem plus a probability problem (random matching / derangements with the surprising exact `Var[Xₙ] = 1` result). Each carries: problem statement, known answer, common failure mode, key-insights checklist. The checklist makes scoring fast for future invocations.
- **Operational constraints baked into the skill text** — "all Agent calls must go out in a single assistant message" (parallel dispatch), "prompt must be byte-identical across models", "self-report is the only in-harness identity check". These came directly from this session's observations.
