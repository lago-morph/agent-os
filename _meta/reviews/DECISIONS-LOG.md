## DECIDED

### Authentication & secrets
- **D-01 — agent auth modes.** Modes are `system` and `system-mediated`; `user-cred` is retired. Agents never hold or manage credentials — the LiteLLM gateway does. System agents that need direct access (e.g. HolmesGPT) use a platform-granted workload identity (service account), separate from `authMode`.
  - **Resolved meanings:** `system` = credentials needed by *system / platform processes* (e.g. HolmesGPT for inference) — **not** tenant-specific. `system-mediated` = credentials the platform (LiteLLM) **holds on behalf of a specific tenant's agents**; LiteLLM enforces per-tenant isolation so an agent's tenant key unlocks only that tenant's credentials, never another's. *How* LiteLLM enforces that (ACLs/keys) is detailed-design. Agents still never hold credentials directly.
- **Secrets abstracted by External Secrets Operator (ESO), delivered in the baseline.** How secrets are encrypted at rest, which KMS is used, and rotation cadence are **out of scope** (baseline / external store).
- **Propagation via reloader.** A reloader-style controller (restart pods when their Secret/ConfigMap changes) is expected in the baseline; if absent, agent-os startup gates on its presence. Chain: external store → ESO updates k8s Secret → reloader restarts consumers.
- **No end-users.** Agents are headless autonomous services. Humans interact only via the approval workflow and operator RBAC/policy. No end-user supplies credentials.
- **LiteLLM credential entry.** LiteLLM reads the credentials it needs from a Secret in its own namespace. Operators may enter credentials (including OAuth credentials — treated as just another secret) through the LiteLLM GUI; LiteLLM writes them to a k8s Secret; ESO **PushSecret** propagates them to the external store. Rule: declare per-secret whether the external store (pull) or LiteLLM/k8s (push) is authoritative — never both on one secret.

### Events, policy ownership, audit, editors
- **D-02 — approval events.** Owned and authored by the approval system (B19), registered into the event catalogue (B12). Build task added to B19.
- **D-03 — OPA decision-document format.** Owned solely by the policy-library framework (B3). OPA (A7) remains the decision engine; the gateway (A1) and sandbox proxy (A6) are consumers of the format, not co-owners.
- **D-05 — audit interface.** Freeze the audit-adapter interface and the `audit_events` schema, with a freeze-gate that must pass before any component emits audit events.
- **D-06 — onboarding editor.** The tenant-onboarding reconciler (A21) owns its own editor; the editor framework (A22) owns the framework + editors for components without their own team.

### Tenancy / quotas
- **#5 — quota enforcement.** `AgentEnvironment` is a Crossplane v2 composite resource (XR), owned by B4, fields `region` / `quotas` / `defaultCapabilitySetRef`. Enforce quotas via **native Kubernetes `ResourceQuota` + `LimitRange`** generated by the composition; admission denies at the Kubernetes API. (The concrete quota field list still needs defining — see #9 / PROPOSED-ADR-A.)
- **#9 — tenant quota is required.** Make quota a **required field** on `TenantOnboarding` at creation. This **reverses** the current A21 / ADR-0037 deferral (quotas were deliberately absent).
- **Quota schema (decided).** v1 ships **native Kubernetes CPU and memory limits per namespace**, enforced via `ResourceQuota` / `LimitRange` emitted by the composition. The schema is an **extensible typed list** (like a Kubernetes typed list / a Pod's `container.env`): `spec.quotas` is a list of entries, each with a `quotaType` discriminator plus type-specific fields, so new quota types can be added later with no breaking change. **Admission rule — mandatory *presence*, not mandatory *limit*:** the `TenantOnboarding` admission controller MUST verify that a `cpu` entry **and** a `memory` entry are present and **positively stated** — each value is either a concrete limit **or** an explicit `unlimited: true`. Omission is rejected; "unlimited" is allowed only when stated outright.

### Policy permissions
- **#23 — distinct permissions.** Provide distinct RBAC permissions at the proposed granularity (e.g. set agent-type default budget vs per-instance override vs budget-maintainer; policy-change vs maintain). Field-level rules native RBAC can't express use Gatekeeper (OPA) admission, already deployed.

### Policy bundle (#24 / #25 / #26 — decided)
- **Ownership (#24):** the **platform security team** owns the global / cross-tenant OPA bundle. Tenants propose changes by pull request; a security reviewer approves before merge.
- **Signing (#25):** policy bundles are **cryptographically signed and verified at load time** — the engine refuses an unsigned or tampered bundle.
- **Staged rollout (#26):** a new/changed bundle first runs in **audit mode** (logs what it *would* block) and flips to **enforce** only after review; the promotion pipeline (Kargo) gates that flip, like it gates code.

### Egress (D-08)
- **Envoy stays.** The Kubernetes agent-sandbox `NetworkPolicySpec` is **L3/L4 only** (IP/CIDR/port, native NetworkPolicy); it cannot do FQDN or HTTP-method allowlisting. The platform needs L7 (`EgressTarget` has `fqdn`/`scheme`/`allowedMethods`), so Envoy remains.
- **Two layers:** sandbox `NetworkPolicySpec` = L3/L4 default-deny baseline for in-cluster traffic; Envoy = L7 (FQDN/method) for external egress, via `EgressTarget` / `CapabilitySet`.
- **Default sandbox egress allow-set — a *minimum baseline*, explicitly extensible (a floor, NOT a closed/mandatory ceiling):** LiteLLM gateway (A1), Envoy egress proxy (A6), audit endpoint (A18), **plus (confirmed):** cluster DNS (CoreDNS), the memory backend (A10), the telemetry/trace collector (A2 / OTel), and the event broker (B8). Additional targets are added as design work surfaces them. Wording must make clear this is a starting set we add to — never imply the list is closed or that membership is mandated/forbidden beyond what's stated.
- **OAuth credentials** are not an end-user flow — operator-entered secrets via the LiteLLM GUI. No OAuth-lifecycle machinery.

### Template layering (confirmed existing)
- Agent "templates" layer via the existing **`CapabilitySet` overlay model** (ADR-0013 / ADR-0032: add-if-not-there / replace-if-there), with per-Agent overrides. B17 is the curated profile library. Egress allowlists (`egressTargets[]`) and other capabilities ride this model — no new template/override mechanism needed.

### Memory store connection (D-04 — resolved as a design-time abstraction)
- Not a now-decision. The **memory backend adapter (B11) owns an abstract memory interface**; each provider (Letta first) is a concrete **subclass** of that interface, and its connection details are **negotiated at design time** with the provider. The connection-secret field list is therefore intentionally **not** pinned in Canon — it lives behind B11's abstraction and can change per provider without disturbing consumers.

### Event-namespace ownership (QN-03 — resolved)
**Principle: exactly one component owns each of the ten `platform.*` CloudEvent namespaces** — it authors and owns the schemas and registers them in the registry (B12). Other components that emit or consume those events take an **explicit dependency** on the owner; never co-ownership.

| Event family (namespace) | Single owner | Who depends |
|---|---|---|
| Agent lifecycle (`platform.lifecycle`) | the agent operator / ARK (A5) | the sandbox (A6), dashboards |
| Gateway (`platform.gateway`) | the LiteLLM gateway (A1) | dashboards, cost reporting |
| Policy (`platform.policy`) | the policy engine / OPA (A7) | dashboards, audit |
| Audit (`platform.audit`) | the audit endpoint (A18) | every audit-emitting component (ties to D-05) |
| Tenant (`platform.tenant`) | the tenant-onboarding reconciler (A21) | dashboards |
| Capability (`platform.capability`) | the kopf operator (B13) — existing | dashboard, SDK, profile library |
| Approval (`platform.approval`) | the approval system (B19) — D-02 | promotion pipeline, dashboard, self-healing agent |
| Security (`platform.security`) | the policy engine / OPA (A7) | dashboards, audit, external SIEM (out of scope) |
| Observability (`platform.observability`) | the metrics & tracing stack — Tempo + Mimir (A13) | external consumers (initially cloud-native logging) |
| Evaluation (`platform.evaluation`) | the test & evaluation framework (B14) | dashboards, promotion pipeline |

**Cross-cutting security requirement (applies to every component):** when a component detects a security-relevant event, it performs its existing local handling **and additionally emits the event to the bus** under `platform.security` (schema owned by A7). Security alerts are treated like policy violations — "we caught something out of the norm worth capturing." The bus emission is an *export/notification* surface (for dashboards, audit, external SIEM), not the primary response path. This needs to land as a requirement in each component spec.

**Observability note (corrected).** AlertManager (which alarms off **Mimir** metrics *and* **Loki** logs) publishes alerts onto the **event bus** under `platform.observability` (A13 owns the schema). Consumers — HolmesGPT and possibly many others — subscribe to alerts **via the bus**; there is **no direct AlertManager→HolmesGPT wire** (point-to-point coupling defeats the bus, whose purpose is emit-once / consumers-choose). Separately, HolmesGPT pulls **metrics from Mimir directly** — a data *stream* it uses to do its job, not events; that must not go on the bus. ⚠️ The corpus currently specifies a direct `AlertManager → HolmesGPT` trigger (glossary / §6.7) — that must be changed to bus-mediated alert consumption.
